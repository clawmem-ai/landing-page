ClawMem automatically manages the full lifecycle of a session — from memory recall at the start, through real-time transcript mirroring, to AI extraction at the end. This guide explains what happens behind the scenes and how to tune the process.

## Overview

Every session goes through these phases:

1. **Memory injection** — Before you start talking, ClawMem recalls relevant memories and injects them into context.
2. **Conversation mirroring** — As you talk, ClawMem mirrors the conversation into a GitHub issue in real time.
3. **AI extraction** — When the session ends (or resets), ClawMem runs an AI extraction pass to decide what memories are worth keeping.
4. **State persistence** — Session state is tracked locally so extraction is idempotent and resumable.

---

## Memory injection at session start

When a new session begins, the `before_agent_start` hook fires. ClawMem:

1. Receives the incoming prompt
2. Searches the agent's default memory repo for relevant memories
3. Retrieves the top N memories (configurable via `memoryRecallLimit`, default 5)
4. Injects them into the agent context as a `<relevant-memories>` block

The agent sees these memories before it processes the prompt, so it starts each conversation already aware of your preferences, constraints, and prior context.

**Configuration:**

```json5
memoryRecallLimit: 5  // Number of memories to recall (1–20)
```

If `memoryRecallLimit` is 0, no memories are recalled. Increase it for broader context; decrease it to reduce noise.

## Session mirroring

As you and the agent exchange messages, ClawMem mirrors the conversation into a GitHub issue in your memory repo.

### How it works

1. Each session gets its own GitHub issue with the title `Session: <auto-generated-or-llm-title>`
2. The issue is tagged `type:conversation` to distinguish it from memory issues
3. Each turn (agent message or tool call) becomes a comment on the issue
4. The issue body tracks metadata: session ID, agent ID, timestamps

### Turn batching with turnCommentDelayMs

Comments are not posted immediately. Instead:

1. When a turn completes, it's queued for mirroring
2. ClawMem waits `turnCommentDelayMs` milliseconds (default 1000ms) before posting the comment
3. If another turn completes during the delay, it's batched into the same comment
4. The comment is finally posted after the delay expires with no new turns

**Why the delay?** Transcript writes often settle asynchronously. A small delay allows multiple updates to accumulate before mirroring, preventing fragmented comments and reducing API calls.

**Configuration:**

```json5
turnCommentDelayMs: 1000  // Delay before posting turn comments (0–60000)
```

Set to 0 for immediate mirroring; increase to 5000+ for slow networks or when batching many turns together.

## AI extraction at session end

At the end of a session — or when the session resets — ClawMem runs an AI extraction pass.

### What extraction does

The extraction subagent:

1. Reads the full session transcript (all turns, all comments)
2. Reads your existing active memories from the memory repo
3. Runs a prompt asking the AI to identify:
   - **save** — New stable, reusable facts, preferences, decisions, constraints, workflows, or context worth remembering
   - **stale** — IDs of existing memories that the current session clearly supersedes or invalidates
4. Outputs a structured JSON decision

### Extraction timing

Extraction is triggered at two points:

**During the session:**
- When the `agent_end` hook fires (end of an agent turn), ClawMem schedules extraction to run after a configurable delay
- This happens in the background; the session continues normally

**At session finalization:**
- When `before_reset` or `session_end` hook fires, ClawMem enqueues a finalization step
- Finalization waits for any in-flight extraction to complete, then writes the final extraction results
- Only after finalization is the session marked as complete

### Extraction timeout

If extraction takes longer than `summaryWaitTimeoutMs`, ClawMem writes a failed extraction record and continues. The session is not blocked.

**Configuration:**

```json5
summaryWaitTimeoutMs: 120000  // Timeout for extraction to complete (1000–600000ms)
```

Default is 2 minutes. Increase if extraction consistently times out; decrease if you have a fast extraction service.

## State persistence

ClawMem tracks session state locally in `.claude/clawmem-state.json`. This state file stores:

```json
{
  "version": 2,
  "sessions": {
    "<scope-key>": {
      "sessionId": "...",
      "sessionKey": "...",
      "agentId": "...",
      "issueNumber": 42,
      "issueTitle": "Session: ...",
      "lastMirroredCount": 5,
      "turnCount": 8,
      "lastMemorySyncCount": 3,
      "summaryStatus": "pending",
      "finalizedAt": "2025-06-15T10:30:00Z",
      "lastSummaryHash": "sha256-...",
      "createdAt": "2025-06-15T09:00:00Z",
      "updatedAt": "2025-06-15T10:30:00Z"
    }
  }
}
```

### State fields

| Field | Purpose |
|---|---|
| `sessionId` | Unique session identifier |
| `sessionKey` | Session subkey (for multi-session scenarios) |
| `agentId` | Which agent the session belongs to |
| `issueNumber` | GitHub issue number for this session |
| `issueTitle` | Title of the session issue |
| `lastMirroredCount` | How many turns have been mirrored to the issue |
| `turnCount` | Total turns in the session |
| `lastMemorySyncCount` | How many memories were synced in the last extraction |
| `summaryStatus` | `pending` or `complete` |
| `finalizedAt` | ISO timestamp when finalization completed |
| `lastSummaryHash` | Hash of the last extraction result (for idempotency) |
| `createdAt`, `updatedAt` | Session timestamps |

---

## Idempotency

ClawMem prevents duplicate memory extraction using a content hash:

1. After extraction completes, a hash of the extraction result is computed
2. This hash is stored in `lastSummaryHash` in the state file
3. On next extraction for the same session, if the new hash matches the old hash, the extraction is skipped
4. This prevents duplicate memory issues from being created if the same session is extracted twice

**Scope key:** Sessions are keyed by `{sessionId}-{sessionKey}`, allowing multiple parallel sessions per agent to be tracked independently.

## Service lifecycle

ClawMem is registered as a service within the OpenClaw runtime.

### Start

When the service starts:

1. Load state from `.claude/clawmem-state.json`
2. Subscribe to `onSessionTranscriptUpdate` events
3. Log the number of configured agent routes
4. Warn if any agent has no default repo configured

### Stop

When the service stops:

1. Unsubscribe from transcript events
2. Clear all pending sync timers
3. Await all in-flight operations to complete
4. Ensure no memory operations are left hanging

---

## Event hooks

ClawMem listens to four OpenClaw lifecycle hooks:

| Hook | When | Purpose |
|---|---|---|
| `before_agent_start` | Before the agent processes a prompt | Recall relevant memories and inject into context |
| `agent_end` | After the agent finishes a turn | Schedule turn mirroring and extraction |
| `before_reset` | Before the session resets or tears down | Finalize extraction, mark session complete |
| `session_end` | When the session formally ends | Finalize extraction, close the session issue |

Additionally, ClawMem subscribes to `onSessionTranscriptUpdate` to listen for real-time transcript changes and mirror them to the session issue.

## Tuning the lifecycle

### Inject more memories at session start

If the agent needs broader context before sessions:

```json5
memoryRecallLimit: 10  // Increase from default 5
```

Range: 1–20. Typical values: 3–10.

### Adjust turn mirroring speed

If transcript comments are arriving too slowly or fragmenting:

```json5
turnCommentDelayMs: 2000  // Increase batching delay
```

If you need real-time mirroring with no batching:

```json5
turnCommentDelayMs: 0  // Post immediately
```

Range: 0–60000ms. Typical values: 500–5000.

### Increase extraction timeout

If extraction is timing out regularly:

```json5
summaryWaitTimeoutMs: 300000  // Increase to 5 minutes
```

Check logs to see how long extraction actually takes, then set the timeout accordingly.

Range: 1000–600000ms. Typical values: 60000–240000.

## Troubleshooting

**Memories are not being recalled**

Check `memoryRecallLimit > 0` and verify memories exist:

```text
memory_list(limit=1)
```

If empty, store a test memory manually.

**Turn comments are not appearing in the session issue**

Check network access and that the transcript is being updated. If `turnCommentDelayMs` is very high, comments may be queued. Decrease the delay to test.

**Extraction is timing out**

Check extraction service logs. If extraction actually completes but slowly, increase `summaryWaitTimeoutMs`. If extraction hangs, check the backend service health.

**Session state file is large or stale**

The state file grows as you accumulate sessions. You can safely delete old entries from `.claude/clawmem-state.json` to clean up. ClawMem will recreate entries as needed.
