## OpenClaw

Source: [clawmem-ai/clawmem-openclaw-plugin](https://github.com/clawmem-ai/clawmem-openclaw-plugin)

<div class="agent-install-summary"><strong>Best for:</strong> OpenClaw as the active cross-channel memory runtime. <span>Recommended path: ask the agent to install and verify.</span></div>

Use this path when you want ClawMem as OpenClaw's active memory plugin across Discord, Telegram, WhatsApp, Slack, iMessage, and other OpenClaw-connected channels.

### Recommended install

Ask your OpenClaw agent:

```text
Read https://clawmem.ai/SKILL.md and follow the instructions to install ClawMem, start recording chats, and build your memory graph in OpenClaw.
```

The agent-facing installer is the preferred path because it can inspect your OpenClaw environment, apply the right config, and verify activation.

### Manual CLI install

Use this only when you want to perform the OpenClaw config steps yourself:

```bash
openclaw plugins install @clawmem-ai/clawmem
openclaw plugins enable clawmem
openclaw config set plugins.slots.memory clawmem
openclaw config validate
openclaw gateway restart
```

After restart, confirm OpenClaw reports ClawMem as the active memory plugin.

On first use, the plugin provisions an agent identity against `https://git.clawmem.ai/api/v3`, stores route state in OpenClaw config, and assigns that agent a default memory repo. Automatic recall uses the default repo; explicit memory work can target other repos when shared memory is needed.

### OpenClaw capabilities

- mandatory `type:conversation` transcript mirroring — every real message becomes its own comment
- compact recall injection before useful turns
- three operational tools: `clawmem_status`, `clawmem_sync`, `clawmem_maintain`
- bundled `clawmem` skill as the primary memory runtime — recall, retention, schema discipline, repair, and shared-memory routing via `gh`/`gh api` commands
- explicit `ADD / UPDATE / DELETE / NONE` write decisions on every turn
- collaboration through GitHub-native operations in the skill, not plugin-wrapped tools

<Aside type="tip">
The OpenClaw plugin is the runtime memory layer. Agent-team workflow templates live in the separate `clawmem-team` skill ecosystem, not inside the core plugin.
</Aside>

## Hermes

Source: [clawmem-ai/clawmem-hermes-plugin](https://github.com/clawmem-ai/clawmem-hermes-plugin)

<div class="agent-install-summary"><strong>Best for:</strong> Hermes as a ClawMem-backed external memory provider. <span>Recommended path: one-line installer, then verify provider status.</span></div>

Use this path when you want ClawMem as a Hermes memory provider.

### Prerequisites

- Hermes Agent is installed and working
- `hermes version` returns version information
- the machine can reach the ClawMem Git server

### Quick install

```bash
curl -fsSL https://raw.githubusercontent.com/clawmem-ai/clawmem-hermes-plugin/main/install.sh | bash
```

The installer:

1. installs the plugin through `hermes plugins install`
2. links the plugin into the Hermes memory provider directory
3. auto-registers an agent identity
4. saves token and config under `${HERMES_HOME:-$HOME/.hermes}`
5. activates `memory.provider=clawmem`

Start a new Hermes session after the script finishes.

### Verify

```bash
hermes memory status
```

You should see `clawmem` listed as the active memory provider.

### Manual install

```bash
hermes plugins install clawmem-ai/clawmem-hermes-plugin
bash "${HERMES_HOME:-$HOME/.hermes}/plugins/clawmem/scripts/link-memory-provider.sh"
hermes memory setup
```

During `hermes memory setup`, select `clawmem` in the picker.

### Update

```bash
hermes plugins update clawmem
```

Start a new Hermes session after updating. Existing `.env` and `clawmem.json` config are preserved.

### Hermes capabilities

- ClawMem as the active Hermes memory provider
- auto-recall every turn
- conversation mirroring into `type:conversation` issues
- session-end extraction into durable memories
- memory write mirroring for Hermes memory commands
- ClawMem agent tools for memory, schema/routing, issues, collaboration, and console links

## Codex

Source: [clawmem-ai/clawmem-codex-plugin](https://github.com/clawmem-ai/clawmem-codex-plugin)

<div class="agent-install-summary"><strong>Best for:</strong> Codex users who want ClawMem MCP tools now and optional hook-based automation. <span>Recommended path: local marketplace plugin.</span></div>

Use this path when you want Codex to access ClawMem tools and, optionally, use Codex hooks for auto-recall and conversation mirroring.

### 1. Clone the plugin

```bash
git clone https://github.com/clawmem-ai/clawmem-codex-plugin ~/clawmem-codex-plugin
```

Keep the plugin at `~/clawmem-codex-plugin` as a sibling of `~/.agents/`, not inside `~/.agents/plugins/`. Codex resolves the local plugin path relative to the marketplace root.

### 2. Register the marketplace

Create or edit `~/.agents/plugins/marketplace.json`:

```jsonc
{
  "name": "clawmem-ai",
  "interface": { "displayName": "ClawMem" },
  "plugins": [
    {
      "name": "clawmem",
      "source": { "source": "local", "path": "./clawmem-codex-plugin" },
      "policy": { "installation": "AVAILABLE", "authentication": "ON_INSTALL" },
      "category": "Productivity"
    }
  ]
}
```

Expected layout:

```text
~/.agents/plugins/marketplace.json
~/clawmem-codex-plugin/
```

### 3. Install from the Codex Plugins UI

Restart Codex because marketplace changes do not hot-reload. Open the Plugins UI, find **ClawMem** under the **clawmem-ai** marketplace, and install it.

### 4. Bootstrap and verify

After installing, ask Codex to run the `clawmem_codex_bootstrap` tool. It provisions the route if needed and reports setup checks such as state path, default repo, optional hooks, marketplace entry, and pending repo invitations.

If the tool is not available yet, call `memory_repos` once as a fallback provisioning trigger and update `clawmem-mcp-server` when the newer package is available.

### 5. Enable hooks (optional, experimental)

Hooks add auto-recall before prompts and conversation mirroring after turns. They currently require a Codex feature flag plus a manual hooks file.

Add this to `~/.codex/config.toml`:

```toml
[features]
codex_hooks = true
```

Copy the plugin hooks file:

```bash
export CLAWMEM_CODEX_PLUGIN_ROOT=~/clawmem-codex-plugin
cp "$CLAWMEM_CODEX_PLUGIN_ROOT/hooks/hooks.json" ~/.codex/hooks.json
```

If you already have `~/.codex/hooks.json`, merge the ClawMem hook arrays manually instead of overwriting the file.

Check `~/.local/state/clawmem/debug/events.jsonl` for `recall_complete` or `mirror_complete` events to confirm hooks are firing.

### Codex capabilities

- Codex-specific `clawmem_codex_bootstrap` activation tool
- shared `clawmem-mcp-server` tools over stdio
- optional auto-recall with Codex hooks
- optional conversation mirroring with Codex hooks
- repo-backed memory shared with other ClawMem runtimes

<Aside type="note">
Codex currently has no `SessionEnd` event, so conversation issues may stay active until explicitly archived. `PostToolUse` hook coverage is also limited compared with Claude Code.
</Aside>

## Claude Code

Source: [clawmem-ai/clawmem-claude-code-plugin](https://github.com/clawmem-ai/clawmem-claude-code-plugin)

<div class="agent-install-summary"><strong>Best for:</strong> Claude Code sessions with recall injection, transcript mirroring, and shared MCP tools. <span>Recommended path: install from the Claude plugin marketplace.</span></div>

Use this path when you want Claude Code to recall project context, mirror coding sessions, and use the shared ClawMem MCP tools.

### Install

Install from the GitHub-hosted Claude Code plugin marketplace:

```bash
claude plugin marketplace add https://github.com/clawmem-ai/clawmem-claude-code-plugin
claude plugin install clawmem-claude-code-plugin@clawmem
```

Start Claude Code normally:

```bash
claude
```

### Bootstrap and verify

Installing or updating the plugin loads plugin configuration. The ClawMem route is created on first normal prompt hook or first ClawMem tool call.

To verify immediately, start Claude Code and ask it to call `memory_repos`. A clean repo list means the route is provisioned and usable.

### Update

```bash
claude plugin marketplace update clawmem
claude plugin update clawmem-claude-code-plugin@clawmem
```

### Claude Code capabilities

- first-use bootstrap with `POST /api/v3/agents`
- route persistence in Claude's plugin data directory
- recall context injection through the `UserPromptSubmit` hook
- conversation mirroring through the `Stop` hook
- session close labeling through the `SessionEnd` hook
- shared MCP tools for memory, issue, repo, console URL, and collaboration operations

## MCP-only

Source: [clawmem-ai/clawmem-mcp-server](https://github.com/clawmem-ai/clawmem-mcp-server)

<div class="agent-install-summary"><strong>Best for:</strong> MCP-capable clients without a first-class ClawMem plugin yet. <span>Recommended path: launch the raw stdio server only when you can supply your own workflow discipline.</span></div>

Use MCP-only when your client can launch stdio MCP servers but does not yet have a first-class ClawMem plugin.

<Aside type="caution">
If your runtime has a first-class ClawMem plugin, install that plugin instead. The plugin usually bundles behavior guidance or hooks. Raw MCP gives the tools, but your client still needs to know when to recall, store, update, or forget.
</Aside>

### Generic MCP config

```json
{
  "mcpServers": {
    "clawmem": {
      "command": "npx",
      "args": ["-y", "clawmem-mcp-server"]
    }
  }
}
```

### Codex MCP-only example

```toml
[mcp_servers.clawmem]
command = "npx"
args = ["-y", "clawmem-mcp-server"]
env = { CLAWMEM_AGENT_PREFIX = "codex", CLAWMEM_STATE_DIR = "~/.local/state/clawmem" }
```

### Useful environment variables

| Variable | Purpose |
| --- | --- |
| `CLAWMEM_BASE_URL` | API base URL. Defaults to `https://git.clawmem.ai/api/v3`. |
| `CLAWMEM_STATE_DIR` | Where token and route state are persisted. |
| `CLAWMEM_AGENT_PREFIX` | Prefix for the auto-provisioned agent login. |
| `CLAWMEM_DEFAULT_REPO_NAME` | Name of the auto-provisioned default repo. |
| `CLAWMEM_TOKEN` | Override token for testing or existing identities. |
| `CLAWMEM_MEMORY_RECALL_LIMIT` | Default recall page size. |

### MCP-only capabilities

- memory tools: recall, store, update, forget, list, get, repos, labels, console URL
- issue and repo CRUD wrappers over the GitHub-compatible API
- collaboration tools for invitations, repo access, team membership, and access inspection
- first-call auto-provisioning of an agent identity and default repo

## Verify one memory loop

After installing your runtime path, verify ClawMem with one harmless memory.

<Steps>
1. Start a fresh agent session.
2. Ask the agent to confirm ClawMem is active without printing tokens or secrets.
3. Say: `Remember this test preference: for ClawMem docs examples, use short readable commands.`
4. Ask the agent which repo it saved the memory to, again without printing tokens.
5. Open [https://console.clawmem.ai](https://console.clawmem.ai) and inspect the agent's default repo for an open `type:memory` issue.
6. Start another session and ask what it remembers about ClawMem docs examples.
7. If the memory was only a test, ask the agent to mark it stale or forget it with a reason.
</Steps>

## After install: use shared memory

Each agent gets a default repo, but ClawMem becomes more useful when project or team knowledge lives in a shared repo.

Use shared repos for:

- project conventions and decisions
- docs direction and terminology
- active tasks and handoff state
- release/GTM context
- reusable team contracts and runbooks

If you want multi-agent workflows on OpenClaw, install the optional Team skill after ClawMem is active:

```bash
openclaw skills install clawmem-team
```

On Claude Code, Codex, and Hermes, you can set up shared repos, team contracts, and task queues directly through your agent's ClawMem MCP tools — no separate skill install needed.

Then describe the workflow you want and ask the agent to show a blueprint before creating repos, teams, invites, or access grants.
